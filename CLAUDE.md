# Surfr Website — Project Rules

## Copy & Writing Style

**NEVER use em-dash stripes (— or --) in customer-facing copy.** They read as AI slop.

Use instead:
- Periods (full stops): "Still the most accurate. Every session captured."
- Commas for tighter flow
- Colons when introducing a list or phrase
- Parentheses when needed for asides
- Line breaks / new sentences for rhythm

Applies to: website copy, headlines, body, bullets, App Store text, release notes, push notifications, emails, social posts. Anything a Surfr user, prospect, or partner reads.

Internal docs and code comments are fine.

## Smartwatch USP

The smartwatch integration is a core differentiator and should be highlighted in feature copy where relevant. Supported platforms: **Apple Watch, Wear OS, and Garmin**. Surfr runs standalone on the watch with live haptics, on-wrist metrics, and post-session sync. Mention the watch (and list all three platforms when space allows) when describing tracking, haptics, or in-session feedback.

## Don't Expose Third-Party Data Sources

Customer-facing copy must never name third-party data providers (Open Meteo, GFS, ICON, ECMWF, AROME, COSMO, ICON-D2, etc.). Use "multiple weather models," "cross-checked forecasts," "high-resolution regional models" instead. Surfr's value is the interpretation, alerting, and cross-checking, not the raw feed. Internal docs and developer-facing pages may name sources where context demands.
